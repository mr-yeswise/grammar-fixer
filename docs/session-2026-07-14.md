## Overview

This document captures the full session: what was broken, what was designed, what was shipped, and what to pick up next time.

**Goal achieved:** Select text in any Windows app ‚Üí press `Ctrl+Alt+G` ‚Üí selected text is corrected and replaced in-place. No full-field replacement.

---

## What Was Broken (Before This Session)

Three compounding bugs caused the app to always replace the entire text field instead of just the selection:

1. **`UiaHelper.GetFocusedText()`** read the entire field ‚Äî no selection awareness at all.
2. **`UiaHelper.SetViaClipboard()`** sent `Ctrl+A` before `Ctrl+V`, which selected all text first ‚Äî nuking the whole document on paste.
3. **`HotkeyManager` modifier reset logic** ‚Äî `_ctrlDown` and `_altDown` were reset on the same keypress that fired the hotkey (`G`), because the reset block ran for every non-modifier key. This meant the modifier state was corrupted exactly when `GetSelectedText()` needed it to be clean.
4. **`CorrectionWindow.GetDiffText()`** returned both old and new diff lines joined together ‚Äî so "Copy Corrected" put corrupted text on the clipboard.
5. **`KeyboardHook`** had no `KeyUp` event ‚Äî only `KeyDown` was exposed, making proper modifier tracking impossible.

## Known Gaps / Next Session

### 1. `App.xaml.cs` CS4014 Warning (Low priority)
An unawaited async call on startup. Harmless but should be cleaned up:
```csharp
// Find the unawaited Task call and prefix with:
_ = SomeAsyncMethod();
// or properly await inside an async method
```

### 2. Electron App Support (Medium priority)
`TextPattern.GetSelection()` does **not** work in Electron apps (VS Code, Slack, Discord). The `ReadSelectionViaClipboard()` fallback handles this, but it has a race condition: if the user's clipboard contained something important and the restore fails (exception between read and restore), they lose it.

**Fix:** Wrap the clipboard save/restore in a proper try/finally in `ReadSelectionViaClipboard()`:
```csharp
private static string? ReadSelectionViaClipboard()
{
    string? prev = null;
    try
    {
        Application.Current.Dispatcher.Invoke(() => prev = Clipboard.GetText());
        SendKeys.SendWait("^c");
        Thread.Sleep(100);
        string? captured = null;
        Application.Current.Dispatcher.Invoke(() => captured = Clipboard.GetText());
        return string.IsNullOrEmpty(captured) ? null : captured;
    }
    finally
    {
        if (!string.IsNullOrEmpty(prev))
            Application.Current.Dispatcher.Invoke(() => Clipboard.SetText(prev));
    }
}
```

### 3. Multi-Selection Support (Low priority)
`TextPattern.GetSelection()` returns an array ‚Äî currently only `ranges[0]` is used. Apps that support discontiguous selection (rare) would only correct the first range.

### 4. Auto-Replace Toggle in Settings (Medium priority)
Currently, `OnHotkeyPressed()` **always** auto-replaces. Some users may want hotkey ‚Üí show correction window for review ‚Üí manually apply. A `Settings.AutoReplaceOnHotkey: bool` toggle would gate this:
```csharp
if (_settings.AutoReplaceOnHotkey)
    UiaHelper.ReplaceSelectedText(result.Corrected);
else
    ShowCorrectionWindow(result);
```

### 5. `_lastWasSelection` Stale State (Medium priority)
The `_lastWasSelection` flag is set at hotkey time and consumed later by `ApplyCorrectionFromWindow()`. If the user presses the hotkey, switches to a different field, then clicks "Replace Selection" in the window ‚Äî it will try to replace a selection that no longer exists. Consider snapshotting the target `AutomationElement` alongside `_lastWasSelection`.
